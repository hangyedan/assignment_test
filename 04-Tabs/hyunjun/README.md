## 💡 설계 및 고민한 부분

> **AI 없이 스스로 설계하며 가장 신경 쓴 핵심 로직이나 아키텍처(컴포넌트 구조, 상태 관리 등), 혹은 사용자 경험(UX) 개선 사항을 적어주세요.**

- 처음에는 `useState` 없이 탭을 구현하려고 했습니다. `data-*` 속성과 `let` 변수를 활용해 현재 선택된 탭 값을 저장하려 했지만, 변수 값이 변경되어도 React가 이를 감지하지 못해 화면이 다시 렌더링되지 않는 문제가 있었습니다. 이 과정에서 React에서 화면에 영향을 주는 값은 상태로 관리해야 한다는 점을 확인했고, 직접 DOM을 조작하기보다 `useState`를 활용하는 방식으로 변경했습니다.

- 이후에는 탭과 탭 콘텐츠를 어떤 구조로 연결할지 고민했습니다. 하나는 `value`, `label`, `description`을 가진 데이터 배열로 관리하고 `map`으로 렌더링하는 방식이었고, 다른 하나는 `children`을 활용해 콘텐츠를 직접 조합할 수 있게 하는 방식이었습니다.

- 사용자가 직접 탭을 추가하는 단순한 구조라면 데이터 배열 방식이 적합하다고 판단했습니다. 실제로 `value`, `label`, `description`을 하나의 객체로 묶어 탭을 구현해보았습니다. 하지만 이 방식은 콘텐츠 형태가 특정 데이터 구조에 묶이기 때문에, 이미지나 복잡한 JSX를 자유롭게 넣기 어렵다는 한계가 있었습니다.

- 그래서 탭 콘텐츠의 자유도를 높이고 재사용성을 개선하기 위해 컴파운드 컴포넌트 패턴으로 재설계했습니다. `Tabs.Trigger`와 `Tabs.Content`를 같은 `value`로 연결하고, 내부 상태는 Context를 통해 공유하도록 구성하여 사용하는 쪽에서 콘텐츠 구조를 더 유연하게 작성할 수 있도록 했습니다.

## 📖 학습한 내용 및 어려웠던 점 (선택)

> **문제를 풀며 새롭게 알게 된 개념이나, 구현 중 막혔던 부분이 있다면 공유해 주세요.**

- 데이터 기반 렌더링 방식과 컴파운드 컴포넌트 패턴의 장단점을 비교해볼 수 있었습니다. 데이터 배열 기반 방식은 구현이 간단하고 관리가 쉽지만, 컴포넌트가 특정 데이터 구조에 의존하게 됩니다. 반면 컴파운드 컴포넌트 패턴은 구현 복잡도는 높지만 JSX를 자유롭게 조합할 수 있어 재사용성과 확장성이 높다는 점을 학습했습니다.

- React 19부터는 `Context`를 사용할 때 기존의 `<Context.Provider>` 대신 `<Context>` 자체를 Provider처럼 사용할 수 있다는 점을 알게 되었습니다.
